# AGENTS.md

This file gives coding agents project-specific instructions for working on
noted. Treat it as the operational guide for this repository.

## Project Overview

noted is a Tauri 2 desktop scratchpad. The frontend is vanilla HTML/CSS/JS
bundled with esbuild, and the backend is Rust exposed through Tauri commands.
Notes are stored locally in SQLite as plain text.

Primary goals:

- Keep the app small, fast, and distraction-free.
- Preserve plain-text note content and local-first behavior.
- Prefer focused changes over broad rewrites.
- Keep release metadata and updater behavior consistent.

## Repository Layout

- `src/` - frontend source loaded into the Tauri webview.
- `src/main.js` - frontend composition root; wires controllers together.
- `src/editor-controller.js` - CodeMirror adapter boundary used by the app.
- `src/notes-controller.js` - note list state, navigation index, save queue, and autosave.
- `src/editor-features/` - CodeMirror state fields, decorations, and editor-only features.
- `src/themes.js` and `src/theme-ui.js` - theme validation, mapping, persistence, and UI.
- `src/updater-ui.js` - frontend updater controls.
- `src/window-controls.js` - custom window controls and quit behavior.
- `src/styles.css` - global app styling and theme token consumption.
- `src-tauri/src/lib.rs` - Tauri setup, commands, plugins, hotkey registration, theme file handling.
- `src-tauri/src/db.rs` - SQLite schema, migrations, note CRUD, and Rust database tests.
- `scripts/build-frontend.mjs` - esbuild bundle step from `src/` to `dist/`.
- `scripts/validate-release.mjs` - release metadata and updater manifest validation.
- `scripts/noted-toggle` - Linux DBus toggle helper installed by Linux packages.
- `docs/` - GitHub Pages website and theme compatibility notes.
- `latest.json` - updater manifest generated by release automation; do not hand-edit for releases.

## Development Commands

Use npm for this repo. There is a `pnpm-lock.yaml`, but the current scripts and
release workflow use `npm install` and `npm run ...`.

Common commands:

```bash
npm install
npm run build:frontend
npm run tauri -- dev
npm run check:rust
npm run test:rust
npm run tauri -- build
```

Release validation:

```bash
npm run validate:release -- --version=v0.2.0
```

When validating changes, choose the smallest meaningful set:

- Frontend-only JS/CSS/HTML: `npm run build:frontend`.
- Rust/Tauri command changes: `npm run check:rust` and `npm run test:rust`.
- Release/version changes: `npm run validate:release -- --version=<tag>` plus the build/check/test commands in the README.
- Shell helper changes: `bash -n scripts/noted-toggle`.

## Architecture Notes

- Keep frontend behavior modular. `src/main.js` should stay as wiring code; put behavior in the relevant controller/module.
- Treat `src/editor-controller.js` as the app-facing editor adapter. Other modules should use its methods instead of reaching into CodeMirror internals.
- Keep editor-only behavior inside `src/editor-features/` when possible.
- Preserve the plain-text storage model. Do not store rich text, HTML, or rendered editor decorations as note content.
- `src/editor-features/mode-detector.js` writes the active mode to CodeMirror's inner `.cm-editor` element with `data-editor-mode`, not to the outer `#note-editor` mount. CSS for mode-specific editor layout should target selectors like `#note-editor .cm-editor[data-editor-mode="list"] ...`.
- Checklist/list rendering is editor-only: `src/editor-features/checklist.js` adds checkbox widgets and marks the rendered `list` keyword, while `src/styles.css` controls their visual spacing. For vertical spacing between checklist rows, prefer list-mode line layout rules in `src/styles.css` rather than changing the global `--editor-line-height`.
- `src/notes-controller.js` serializes saves with `pendingSave`; do not bypass this queue when adding note operations.
- `src-tauri/src/db.rs` owns persistence invariants: at least one note exists, positions remain contiguous, and old schemas migrate forward.
- `src-tauri/src/lib.rs` owns Tauri command registration, plugin setup, app data paths, DBus/single-instance behavior, and theme file commands.
- Imported themes are saved as JSON files under the app data `themes/` directory after validation.
- Linux global hotkey behavior is desktop-owned. The packaged `noted-toggle` script calls the running app over DBus and falls back to launching `noted --toggle`.

## Code Style

- Use ES modules in frontend files.
- Prefer small factory functions returning explicit method objects, matching the existing controller style.
- Keep comments sparse and useful; explain non-obvious constraints, not simple assignments.
- Use ASCII text in source/docs unless a file already needs non-ASCII content.
- Keep CSS tied to the existing theme custom properties instead of hardcoding one-off colors.
- In Rust, keep command functions thin and push persistence rules into `db.rs` or focused helpers.
- Do not add a frontend framework unless explicitly requested.

## Testing Guidelines

- Add or update Rust tests for persistence, migrations, theme validation, or Tauri-adjacent pure helpers.
- For frontend behavior, at minimum run `npm run build:frontend`; add targeted tests only if a test harness is introduced.
- If touching navigation, autosave, delete behavior, or window close/quit paths, reason about pending saves and verify the relevant flow manually or with focused tests.
- If touching updater/release code, run `npm run validate:release -- --version=<tag>` and check that `latest.json` validation still matches the intended release process.

## Release Rules

- Source versions must stay aligned across:
  - `package.json`
  - `package-lock.json`
  - `src-tauri/Cargo.toml`
  - `src-tauri/tauri.conf.json`
  - visible version metadata in `src/index.html`
  - website structured data in `docs/index.html`
- `latest.json` contains signed artifact metadata. For a new release, let the GitHub Actions release workflow generate it from the built artifacts.
- Releases are tag-driven. After version bumps pass validation, push `main` and the `v*` tag.
- Keep README release commands in sync with the current intended release version when doing version-release work.

## Security And Safety

- Never commit secrets, private signing keys, tokens, or local credentials.
- Do not weaken theme JSON validation to accept arbitrary scripts or non-color values.
- Do not add shell execution paths from note content, theme content, or untrusted user input.
- Be careful with filesystem writes: theme files should stay under the app data theme directory, and note data should stay in the app data database.
- Avoid destructive git commands unless the user explicitly asks for them.

## Working Tree Discipline

- The user may have unrelated local changes. Inspect `git status --short` before staging or committing.
- Stage only files relevant to the requested change.
- Generated or local media files should not be added unless the user explicitly asks.
- Do not revert unrelated edits.

## Documentation Guidance

- Keep `README.md` product-facing and concise.
- Keep implementation details that are mainly for agents in this file.
- Keep theme compatibility specifics in `docs/antinote-theme-compatibility.md`.
- If shipped behavior changes, update README or docs in the same change when user-facing claims would become stale.
